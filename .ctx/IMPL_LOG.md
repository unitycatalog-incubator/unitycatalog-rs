## 2026-03-29 â€” Remove `Principal::Custom` and Decouple Principal from Public API

**Motivation:** An auth review surfaced two issues:
1. `Principal::Custom(Bytes)` is an opaque extension mechanism that overlaps with the `Cx` type
   parameter â€” the intended ABAC extension point. No production code constructs it; it appeared
   only in one test match arm. Dead API surface with no users.
2. `Principal` was tightly coupled to the public framework contract: the `Authenticator` trait
   hardcoded `-> Result<Principal>`, forcing downstream crates to use `Principal` even if they
   wanted a different identity type (e.g. `JwtClaims`). Making the middleware generic over the
   identity type lets downstream crates stay entirely decoupled from `Principal`.

**Files changed:**

- **`crates/server/src/policy/mod.rs`** â€” Removed `Custom(Bytes)` variant, its constructor, and
  the `use bytes::Bytes` import. The `bytes` dependency stays in `Cargo.toml` (used elsewhere).

- **`crates/server/src/rest/auth.rs`** â€” `Authenticator<I>` is now generic over
  `I: Clone + Send + Sync + 'static`. `AuthenticationMiddleware<S, T, I = Principal>` and
  `AuthenticationLayer<T, I = Principal>` carry a defaulted `I` so all existing call sites
  (`AuthenticationLayer::new(authenticator)`) compile unchanged. `AnonymousAuthenticator`
  implements `Authenticator<Principal>`. Test updated to remove `Custom(_)` arm.

- **`crates/server/src/sharing/codegen/handler.rs`** â€” Added `<Cx = RequestContext>` to
  `SharingHandler`; all methods use `context: Cx`.

- **`crates/server/src/api/sharing/handler.rs`** â€” Added `<Cx = RequestContext>` to
  `SharingQueryHandler`; blanket `SharingHandler<Cx>` impl generalized to
  `impl<T, Cx> SharingHandler<Cx> for T where T: ResourceStore + Policy<Cx> + ShareHandler<Cx>,
  Cx: Clone + Send + Sync + 'static`. The `Cx: Clone` bound is needed because `list_shares`
  recurses for empty pages and passes `context` by value.

- **`crates/server/src/rest/routers/sharing.rs`** â€” `get_router` and all handler fns made generic
  over `<T, Cx>`. `Extension<Principal>` replaced with `context: Cx` extracted via
  `FromRequestParts`. `use crate::policy::Principal` removed entirely.

- **`crates/server/src/sharing/codegen/mod.rs`** â€” Removed stale
  `//! AUTOGENERATED FILE - DO NOT EDIT` header (see follow-up below).

- **`crates/cli/src/server/run.rs`** â€” `run_server_rest` bounds updated to `SharingHandler<Cx>`,
  `SharingQueryHandler<Cx>`, `Authenticator<unitycatalog_server::policy::Principal>`.

**Key design decisions:**

- Default type params (`I = Principal`) were chosen over a separate `AnonymousAuthenticator`-
  specific layer type so that the one-liner `AuthenticationLayer::new(auth)` keeps working. A
  downstream crate wanting `JwtClaims` would write `AuthenticationLayer::<_, JwtClaims>::new(auth)`.

- `SharingQueryHandler` return types (`-> Result<Bytes>` for `get_table_metadata` / `query_table`)
  were left unchanged. The codegen trait (`SharingHandler`) uses `-> Result<QueryResponse>` for
  those slots because the codegen doesn't know about the streaming response shape; the
  `SharingQueryHandler` trait is the hand-written companion that handles the actual wire format.
  These two traits serve different purposes and were kept as-is.

- Non-sharing blanket impls (catalogs, schemas, tables, etc.) were intentionally left as
  `impl<T: ResourceStore + Policy<RequestContext>> FooHandler<RequestContext> for T`. Generalizing
  them to `<Cx>` is a larger change tracked as a follow-up.

**Verification:** `cargo check --workspace` clean; `cargo test -p unitycatalog-server -p unitycatalog-cli` all pass.

## Follow-up Work Identified

### 1. Wire sharing service proto into the build pipeline
**Context:** `crates/server/src/sharing/codegen/handler.rs` has a stale "AUTOGENERATED" header
and is **not currently regenerated by any pipeline**. The `generate-code` justfile recipe only
processes `proto/unitycatalog`; `buf.gen.sharing.yaml` only generates model/client code via
`neoeinstein-prost`. The sharing handler was generated historically and has been maintained by
hand since.

**What needs doing:** Extend the `generate-code` recipe (or add a separate `generate-sharing-code`
recipe) to run the build crate against `proto/sharing` descriptors and write the output to
`crates/server/src/sharing/codegen/`. The build crate already supports a configurable
`context_type_path`, `result_type_path`, and `models_path_template`, so this should be a
justfile + descriptor-build change plus wiring the correct model path
(`unitycatalog_sharing_client::models::sharing::v1`).

### 2. Generalize non-sharing blanket impls over `<Cx>`
**Context:** All non-sharing handler blanket impls (catalogs, schemas, tables, credentials, etc.)
are hardcoded to `impl<T: ResourceStore + Policy<RequestContext>> FooHandler<RequestContext> for T`.
The same `<Cx>` generalization applied to the sharing handler here should be applied uniformly.
This would let downstream crates implement handlers with a custom context type end-to-end without
touching `RequestContext` at all.

### 3. Implement `ReverseProxyAuthenticator`
**Context:** `AnonymousAuthenticator` has a TODO comment since the initial implementation:
implement a `ReverseProxyAuthenticator` that reads `X-Forwarded-User` (or similar) from the
upstream proxy and returns `Principal::User(name)`. Now that `Authenticator<I>` is generic, a
`ReverseProxyAuthenticator` implementing `Authenticator<Principal>` is the natural next step for
production deployments.
