The policy layer determines whether a given principal is allowed to perform an action on a resource. It sits between request handling and the storage layer.

<Aside type="caution">
`ConstantPolicy` (allow-all) is the only built-in implementation. Role-based access control (RBAC) is planned but not yet available. See [Current Status](#current-status) for details.
</Aside>

## `Principal`

Represents the identity making a request.

```rust
pub enum Principal {
    Anonymous,
    User(String),
    Custom(Bytes),
}
```

| Variant | Description |
|---------|-------------|
| `Anonymous` | No authenticated identity; used when no credentials are present |
| `User(String)` | A named user identity, e.g. from a reverse proxy header |
| `Custom(Bytes)` | Raw byte payload for custom authentication schemes |

### Constructor Methods

| Method | Signature | Description |
|--------|-----------|-------------|
| `anonymous` | `fn anonymous() -> Self` | Creates an `Anonymous` principal |
| `user` | `fn user(name: impl Into<String>) -> Self` | Creates a `User` principal |
| `custom` | `fn custom(data: Bytes) -> Self` | Creates a `Custom` principal |

## `Permission`

The access level required to perform an operation. Serialized as `snake_case`.

```rust
pub enum Permission {
    Read,
    Write,
    Manage,
    Create,
    Use,
    Browse,
    Select,
}
```

| Variant | Typical use |
|---------|-------------|
| `Read` | List and get operations |
| `Write` | Modify data within an existing resource |
| `Manage` | Update or delete a resource's metadata |
| `Create` | Create new resources |
| `Use` | Use a resource as part of another operation (e.g. attach a credential) |
| `Browse` | Discover a resource without reading its contents |
| `Select` | Query data from a table or volume |

`Permission` implements `AsRefStr`, `EnumString` (case-insensitive), and `From<Permission> for String`.

## `Decision`

The outcome of a policy check.

```rust
pub enum Decision {
    Allow,
    Deny,
}
```

| Variant | Meaning |
|---------|---------|
| `Allow` | The action is permitted |
| `Deny` | The action is forbidden |

## `Policy` Trait

The core authorization interface. All methods are `async`.

```rust
#[async_trait]
pub trait Policy: Send + Sync + 'static { ... }
```

### Methods

#### `authorize` (required)

```rust
async fn authorize(
    &self,
    resource: &ResourceIdent,
    permission: &Permission,
    recipient: &Principal,
) -> Result<Decision>
```

The only method that must be implemented. Returns `Decision::Allow` if `recipient` holds `permission` on `resource`, otherwise `Decision::Deny`.

#### `check`

```rust
async fn check(&self, obj: &dyn SecuredAction, recipient: &Principal) -> Result<Decision>
```

Convenience wrapper. Calls `authorize` using `obj.resource()` and `obj.permission()`.

#### `check_required`

```rust
async fn check_required(&self, obj: &dyn SecuredAction, recipient: &Principal) -> Result<()>
```

Like `check`, but maps `Decision::Deny` to `Err(Error::NotAllowed)`. Use this when the caller needs to propagate a denial as an error.

#### `authorize_many`

```rust
async fn authorize_many(
    &self,
    resources: &[ResourceIdent],
    permission: &Permission,
    recipient: &Principal,
) -> Result<Vec<Decision>>
```

Checks the same permission against a slice of resources and returns one `Decision` per resource, in order. The default implementation calls `authorize` sequentially; implementations may override for batched evaluation.

#### `authorize_checked`

```rust
async fn authorize_checked(
    &self,
    resource: &ResourceIdent,
    permission: &Permission,
    recipient: &Principal,
) -> Result<()>
```

Like `authorize`, but maps `Decision::Deny` to `Err(Error::NotAllowed)`.

## `ProvidesPolicy`

A marker trait for types that own a `Policy`. Implementing this trait makes the type itself act as a `Policy` through a blanket impl.

```rust
pub trait ProvidesPolicy: Send + Sync + 'static {
    fn policy(&self) -> &Arc<dyn Policy>;
}
```

All `Policy` method calls on a `ProvidesPolicy` implementor delegate to `self.policy()`.

## `SecuredAction` Trait

Implemented by request types that carry both a target resource and a required permission.

```rust
pub trait SecuredAction: Send + Sync {
    fn resource(&self) -> ResourceIdent;
    fn permission(&self) -> &'static Permission;
}
```

| Method | Description |
|--------|-------------|
| `resource()` | Returns the `ResourceIdent` the action targets |
| `permission()` | Returns a reference to the `Permission` needed for this action |

Handler types in `crates/server/src/api/` implement `SecuredAction` for each request variant, providing the per-endpoint permission mapping that drives `Policy::check`.

## `process_resources`

Filters a mutable `Vec` of resources in place, retaining only those for which the policy returns `Decision::Allow`.

```rust
pub async fn process_resources<T: Policy + Sized, R: ResourceExt + Send>(
    handler: &T,
    recipient: &Principal,
    permission: &Permission,
    resources: &mut Vec<R>,
) -> Result<()>
```

Internally calls `authorize_many` for all resources, then retains entries whose corresponding decision is `Allow`. Used by list endpoints to filter results based on the caller's permissions before returning them.

## `ConstantPolicy`

A policy that always returns the same `Decision` regardless of resource, permission, or principal.

```rust
pub struct ConstantPolicy { /* decision: Decision */ }
```

| Method | Description |
|--------|-------------|
| `ConstantPolicy::default()` | Returns a policy that always allows (`Decision::Allow`) |
| `ConstantPolicy::new(decision)` | Returns a policy with the specified constant decision |

`ConstantPolicy` implements `Policy`, `Clone`, `Copy`, and `Debug`. It is `Send + Sync`.

### When to use

- **Development and testing**: `ConstantPolicy::default()` (allow-all) removes authorization from the critical path so you can focus on other behavior.
- **Deny-all testing**: `ConstantPolicy::new(Decision::Deny)` verifies that your handler correctly propagates `Error::NotAllowed`.
- **Trusted internal deployments**: When the server is deployed behind a network boundary that already enforces access control, `ConstantPolicy::default()` avoids redundant checks.

Do not use `ConstantPolicy::default()` in production environments where multi-tenant or user-scoped authorization is required.

## Current Status

`ConstantPolicy` is the only policy implementation shipped with the server. The `Principal` variants and `Permission` enum define the full intended surface for future implementations. RBAC support — where permissions are stored per-resource and evaluated against user or group identity — is planned.

To integrate a custom policy, implement the `Policy` trait (or wrap your state in a struct that implements `ProvidesPolicy`) and supply it when constructing the server.
