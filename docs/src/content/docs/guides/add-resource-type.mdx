---
title: Add a New Resource Type
description: Recipe for adding a new resource type through the proto → codegen → handler implementation pipeline.
---

This guide walks a contributor through the full pipeline for adding a new first-class resource type to the server. Use `widgets` as a placeholder name throughout; substitute your actual resource name.

## Prerequisites

- Rust 1.85+
- `buf` CLI installed (`brew install bufbuild/buf/buf` or see [buf.build/docs](https://buf.build/docs/installation))
- `just` task runner installed

## Overview of generated vs hand-written files

| File | Origin |
|------|--------|
| `crates/common/src/models/gen/unitycatalog.widgets.v1.rs` | Generated by `buf` from proto |
| `crates/server/src/codegen/widgets.rs` | Generated by `just generate-code` |
| `crates/server/src/api/widgets.rs` | **Hand-written** — your business logic |
| `crates/server/src/rest/routers/mod.rs` | **Hand-written** — router registration |
| `crates/cli/src/server/run.rs` | **Hand-written** — handler wiring |

## 1. Add the proto definition

Create two files under `proto/unitycatalog/unitycatalog/widgets/v1/`:

**`models.proto`** — define the resource message:

```protobuf
syntax = "proto3";

package unitycatalog.widgets.v1;

import "google/api/field_behavior.proto";
import "google/api/resource.proto";

message Widget {
  option (google.api.resource) = {
    type: "unitycatalog.io/Widget"
    pattern: "widgets/{widget}"
    plural: "widgets"
    singular: "widget"
  };

  string name = 1;
  optional string id = 2;
  optional string comment = 3;
}
```

**`svc.proto`** — define request/response messages and the service:

```protobuf
syntax = "proto3";

package unitycatalog.widgets.v1;

import "buf/validate/validate.proto";
import "gnostic/openapi/v3/annotations.proto";
import "google/api/annotations.proto";
import "google/api/field_behavior.proto";
import "google/protobuf/empty.proto";
import "unitycatalog/widgets/v1/models.proto";

message ListWidgetsRequest {
  optional int32 max_results = 1 [(google.api.field_behavior) = OPTIONAL];
  optional string page_token = 2 [(google.api.field_behavior) = OPTIONAL];
}

message ListWidgetsResponse {
  repeated Widget widgets = 1;
  optional string next_page_token = 2;
}

message CreateWidgetRequest {
  string name = 1 [
    (buf.validate.field).string.pattern = "^[a-z][0-9a-z_]*[0-9a-z]$",
    (buf.validate.field).string.min_len = 3,
    (google.api.field_behavior) = REQUIRED
  ];
  optional string comment = 2 [(google.api.field_behavior) = OPTIONAL];
}

message GetWidgetRequest {
  string name = 1 [(google.api.field_behavior) = REQUIRED];
}

message UpdateWidgetRequest {
  string name = 1 [(google.api.field_behavior) = REQUIRED];
  optional string comment = 2 [(google.api.field_behavior) = OPTIONAL];
}

message DeleteWidgetRequest {
  string name = 1 [(google.api.field_behavior) = REQUIRED];
}

service WidgetsService {
  rpc ListWidgets(ListWidgetsRequest) returns (ListWidgetsResponse) {
    option (google.api.http) = {get: "/widgets"};
    option (gnostic.openapi.v3.operation) = {operation_id: "ListWidgets"};
  }
  rpc CreateWidget(CreateWidgetRequest) returns (Widget) {
    option (google.api.http) = {post: "/widgets" body: "*"};
    option (gnostic.openapi.v3.operation) = {operation_id: "CreateWidget"};
  }
  rpc GetWidget(GetWidgetRequest) returns (Widget) {
    option (google.api.http) = {get: "/widgets/{name}"};
    option (gnostic.openapi.v3.operation) = {operation_id: "GetWidget"};
  }
  rpc UpdateWidget(UpdateWidgetRequest) returns (Widget) {
    option (google.api.http) = {patch: "/widgets/{name}" body: "*"};
    option (gnostic.openapi.v3.operation) = {operation_id: "UpdateWidget"};
  }
  rpc DeleteWidget(DeleteWidgetRequest) returns (google.protobuf.Empty) {
    option (google.api.http) = {delete: "/widgets/{name}"};
    option (gnostic.openapi.v3.operation) = {operation_id: "DeleteWidget"};
  }
}
```

## 2. Register the new proto with buf

The `buf.yaml` at the repo root already includes `proto/unitycatalog` as a module. No changes needed there.

The `buf.gen.yaml` at the repo root controls code generation targets. It already emits Rust types from all protos under `proto/unitycatalog`, so no changes are required there either.

## 3. Run the full codegen pipeline

```bash
just generate
```

This runs `buf generate` (producing Rust types in `crates/common/src/models/gen/`) followed by the internal codegen binary (producing handler scaffolding in `crates/server/src/codegen/`). Check the output for any protobuf validation errors before proceeding.

## 4. Add the `ObjectLabel` variant

The `ObjectLabel` enum is derived from the `Resource` oneof in `proto/unitycatalog/unitycatalog/internal/resources.proto`. Add a `Widget` variant to the `Resource` message's oneof so the store can address widget resources. Then re-run `just generate`.

## 5. Implement the handler in `crates/server/src/api/widgets.rs`

Create a new file. The generated `WidgetHandler` trait in `crates/server/src/codegen/widgets.rs` defines the method signatures. Implement it on any type that satisfies `ResourceStore + Policy`:

```rust
use unitycatalog_common::models::widgets::v1::*;
use unitycatalog_common::models::{ObjectLabel, ResourceIdent, ResourceName, ResourceRef};

use super::{RequestContext, SecuredAction};
pub use crate::codegen::widgets::WidgetHandler;
use crate::policy::{Permission, Policy, process_resources};
use crate::store::ResourceStore;
use crate::Result;

#[async_trait::async_trait]
impl<T: ResourceStore + Policy> WidgetHandler for T {
    async fn list_widgets(
        &self,
        request: ListWidgetsRequest,
        context: RequestContext,
    ) -> Result<ListWidgetsResponse> {
        self.check_required(&request, context.as_ref()).await?;
        let (mut resources, next_page_token) = self
            .list(
                &ObjectLabel::Widget,
                None,
                request.max_results.map(|v| v as usize),
                request.page_token,
            )
            .await?;
        process_resources(self, context.as_ref(), &Permission::Read, &mut resources).await?;
        Ok(ListWidgetsResponse {
            widgets: resources
                .into_iter()
                .map(|r| r.try_into())
                .collect::<std::result::Result<_, _>>()?,
            next_page_token,
        })
    }

    // ... implement create_widget, get_widget, update_widget, delete_widget
}

impl SecuredAction for ListWidgetsRequest {
    fn resource(&self) -> ResourceIdent {
        ResourceIdent::widget(ResourceRef::Undefined)
    }
    fn permission(&self) -> &'static Permission {
        &Permission::Read
    }
}

impl SecuredAction for CreateWidgetRequest {
    fn resource(&self) -> ResourceIdent {
        ResourceIdent::widget(ResourceName::new([self.name.as_str()]))
    }
    fn permission(&self) -> &'static Permission {
        &Permission::Create
    }
}

// Implement SecuredAction for Get, Update, Delete requests similarly.
```

Follow the pattern used in `crates/server/src/api/credentials.rs` for reference.

## 6. Register the router in `crates/server/src/rest/routers/mod.rs`

Add a new `create_widgets_router` function alongside the existing ones:

```rust
pub fn create_widgets_router<T: WidgetHandler + Clone>(handler: T) -> axum::Router {
    use crate::codegen::widgets::server::*;

    axum::Router::new()
        .route("/widgets", get(list_widgets::<T>))
        .route("/widgets", post(create_widget::<T>))
        .route("/widgets/{name}", get(get_widget::<T>))
        .route("/widgets/{name}", patch(update_widget::<T>))
        .route("/widgets/{name}", delete(delete_widget::<T>))
        .with_state(handler)
}
```

## 7. Wire the handler into `run_server_rest` in `crates/cli/src/server/run.rs`

Add `WidgetHandler` to the trait bounds on `T` and merge the new router into `api_routes`:

```rust
// Add to the where clause:
+ WidgetHandler

// Add to api_routes:
.merge(create_widgets_router(handler.clone()))
```

## 8. Export the handler from `crates/server/src/api/mod.rs`

```rust
pub use widgets::WidgetHandler;
pub mod widgets;
```

## 9. Write a test

Add a journey in `crates/acceptance/src/journeys/` following the pattern of `catalog_simple.rs`:

1. Create a struct implementing `UserJourney`.
2. In `execute`, use `UnityCatalogClient` to create, get, list, and delete a widget.
3. Register the journey in `crates/acceptance/src/journeys/mod.rs` by adding it to `all_journeys()`.
4. Record responses: `just integration-record` (requires a live server).
5. Verify replay: `just integration`.

Run unit tests with:

```bash
cargo test --workspace
```
