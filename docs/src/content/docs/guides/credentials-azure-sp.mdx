---
title: Configure Azure Service Principal Credentials
description: Step-by-step recipe for creating an Azure Service Principal credential and attaching it to an external location.
---

This guide shows how to create an Azure Service Principal (SP) storage credential and attach it to an external location. It assumes the UC server is already running.

## Prerequisites

- A running Unity Catalog server (see `just rest` to start one locally)
- An Azure subscription with an app registration and client secret
- `curl` or the Python client

## 1. Create the credential

Send a `POST` to `/api/2.1/unity-catalog/credentials`. The `name` field must match the pattern `^[a-z][0-9a-z_]*[0-9a-z]$` and be at least 3 characters.

```bash
curl -s -X POST http://localhost:8080/api/2.1/unity-catalog/credentials \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my_azure_sp_cred",
    "purpose": "STORAGE",
    "comment": "Azure SP for ADLS Gen2 access",
    "azure_service_principal": {
      "directory_id": "<AAD_TENANT_ID>",
      "application_id": "<APP_CLIENT_ID>",
      "client_secret": "<APP_CLIENT_SECRET>"
    }
  }'
```

The server returns the created `Credential` object. Note the `name` field — you will reference it when creating the external location.

### Using a federated token instead of a client secret

Replace `client_secret` with `federated_token_file` if you are using workload identity federation:

```json
"azure_service_principal": {
  "directory_id": "<AAD_TENANT_ID>",
  "application_id": "<APP_CLIENT_ID>",
  "federated_token_file": "/var/run/secrets/azure/tokens/azure-identity-token"
}
```

## 2. Create an external location

Send a `POST` to `/api/2.1/unity-catalog/external-locations`. Set `credential_name` to the name you used above.

```bash
curl -s -X POST http://localhost:8080/api/2.1/unity-catalog/external-locations \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my_adls_location",
    "url": "abfss://mycontainer@mystorageaccount.dfs.core.windows.net/",
    "credential_name": "my_azure_sp_cred",
    "comment": "ADLS Gen2 container for raw data"
  }'
```

## 3. Verify the external location

Retrieve the location to confirm it was created and linked to the credential:

```bash
curl -s http://localhost:8080/api/2.1/unity-catalog/external-locations/my_adls_location
```

The response includes `credential_id` (the internal ID of the associated credential) and `credential_name`.

List all external locations to confirm it appears:

```bash
curl -s http://localhost:8080/api/2.1/unity-catalog/external-locations
```

## 4. Rotate the credential

When the client secret expires, update the credential with a new secret via `PATCH`:

```bash
curl -s -X PATCH \
  http://localhost:8080/api/2.1/unity-catalog/credentials/my_azure_sp_cred \
  -H "Content-Type: application/json" \
  -d '{
    "azure_service_principal": {
      "directory_id": "<AAD_TENANT_ID>",
      "application_id": "<APP_CLIENT_ID>",
      "client_secret": "<NEW_CLIENT_SECRET>"
    }
  }'
```

Pass `"force": true` to overwrite even when dependent external locations exist.

## 5. Clean up

Delete the external location first, then the credential (the credential cannot be deleted while an external location references it):

```bash
curl -s -X DELETE \
  http://localhost:8080/api/2.1/unity-catalog/external-locations/my_adls_location

curl -s -X DELETE \
  http://localhost:8080/api/2.1/unity-catalog/credentials/my_azure_sp_cred
```
