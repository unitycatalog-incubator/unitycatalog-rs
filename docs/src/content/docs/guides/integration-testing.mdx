---
title: Run Integration Tests
description: How to run the acceptance test suite in both live-server and recorded-response modes.
---

The acceptance test suite lives in `crates/acceptance/`. It is structured as a set of *user journeys* — each journey creates real resources, inspects them, and cleans up. Journeys can run against a live server (recording mode) or against saved HTTP recordings (replay mode).

## Run with recorded responses (no live server needed)

Recordings are stored in `crates/acceptance/recordings/` and committed to the repository. By default, `just integration` runs in replay mode: it spins up a mock HTTP server from the saved JSON files and executes the journeys against it.

```bash
just integration
```

No environment variables are required. The command runs:

```
UC_INTEGRATION_DIR=<repo>/crates/acceptance/recordings
UC_INTEGRATION_RECORD=false
cargo run --bin unitycatalog-acceptance
```

Each journey directory under `recordings/` contains numbered JSON files (`0000.json`, `0001.json`, …) capturing request/response pairs in order, plus a `journey_state.json` file with any state the journey saves between steps (for example, the catalog name generated at record time).

## Run against a live server

### Start the server

```bash
just rest
```

The server listens on `http://localhost:8080` by default.

### Record new responses

```bash
UC_INTEGRATION_URL=http://localhost:8080 \
UC_INTEGRATION_TOKEN=<optional-token> \
UC_INTEGRATION_DIR=<repo>/crates/acceptance/recordings \
UC_INTEGRATION_STORAGE_ROOT=s3://your-bucket/ \
UC_INTEGRATION_RECORD=true \
cargo run --bin unitycatalog-acceptance
```

Or use the `just` recipe, which expects `$DATABRICKS_HOST`, `$DATABRICKS_TOKEN`, and `$DATABRICKS_STORAGE_ROOT` in your environment:

```bash
just integration-record
```

After a successful run, commit the updated files under `crates/acceptance/recordings/`.

## Run Node.js tests

### Unit tests (no server needed)

```bash
just test-node
```

This runs `npm run test -w @unitycatalog/client` inside the `node/client/` package.

### Integration tests (server started automatically)

```bash
just test-node-integration
```

This builds the Node.js bindings and then runs `npm run test:integration -w @unitycatalog/client`, which starts its own UC server process internally.

## Run Rust unit tests

```bash
cargo test --workspace
```

Run tests for a specific crate only:

```bash
cargo test -p unitycatalog-server
cargo test -p unitycatalog-acceptance
```

## Add a new integration test

1. **Create a journey struct** in a new file under `crates/acceptance/src/journeys/`:

   ```rust
   pub struct MyResourceJourney {
       resource_name: String,
   }

   #[async_trait]
   impl UserJourney for MyResourceJourney {
       fn name(&self) -> &str { "my_resource" }
       fn description(&self) -> &str { "Create, inspect, and delete a my_resource" }

       async fn execute(&self, client: &UnityCatalogClient) -> AcceptanceResult<()> {
           // use client to call your new API endpoints
           Ok(())
       }
   }
   ```

   See `crates/acceptance/src/journeys/catalog_simple.rs` for a complete example, including how to use `JourneyLogger` for structured step output and `save_state`/`load_state` for persisting data between record and replay runs.

2. **Register the journey** in `crates/acceptance/src/journeys/mod.rs`:

   ```rust
   mod my_resource;
   pub use my_resource::MyResourceJourney;

   pub fn all_journeys() -> Vec<Box<dyn UserJourney>> {
       vec![
           Box::new(CatalogSimpleJourney::new()),
           Box::new(CatalogHierarchyJourney::new()),
           Box::new(MyResourceJourney::new()), // add here
       ]
   }
   ```

3. **Record responses** against a live server:

   ```bash
   just integration-record
   ```

   This creates `crates/acceptance/recordings/my_resource/0000.json`, `0001.json`, and so on.

4. **Verify replay** works:

   ```bash
   just integration
   ```

5. Commit the journey source file and the recordings together.
