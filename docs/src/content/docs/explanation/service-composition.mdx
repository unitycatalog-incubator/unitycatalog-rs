title: Service Composition
description: >
  How ServerHandler composes Policy, ResourceStore, and SecretManager via
  Arc wrapping and the Provides* compile-time dependency-injection pattern.
---

`ServerHandler` is the central type that Axum route handlers receive as shared
state. It is cheap to clone (two `Arc` bumps), safe to share across threads, and
extensible to new backends without changes to any handler code. This page
explains how that composition is structured.

## The two-layer structure

```rust
#[derive(Clone)]
pub struct ServerHandler {
    handler: Arc<ServerHandlerInner>,
    session: Arc<KernelSession>,
}

#[derive(Clone)]
pub struct ServerHandlerInner {
    policy:  Arc<dyn Policy>,
    store:   Arc<dyn ResourceStore>,
    secrets: Arc<dyn SecretManager>,
}
```

`ServerHandler` wraps two `Arc` values. `Arc<ServerHandlerInner>` holds the
three service dependencies as trait objects. `Arc<KernelSession>` wraps the
delta-kernel session used for snapshot reads (see the `TableManager` section
below). Cloning a `ServerHandler` increments two reference counts and nothing
else; there is no deep copy.

The split between `ServerHandler` and `ServerHandlerInner` exists because
`KernelSession` needs a reference to `ServerHandlerInner` during construction —
it requires access to the store and secrets to resolve storage locations. By
constructing `ServerHandlerInner` first and wrapping it in its own `Arc`, the
session can hold a weak reference without a circular ownership cycle.

## Construction

```rust
impl ServerHandler {
    pub fn try_new_tokio(
        policy:  Arc<dyn Policy>,
        store:   Arc<dyn ResourceStore>,
        secrets: Arc<dyn SecretManager>,
    ) -> Result<Self> {
        let handler = Arc::new(ServerHandlerInner::new(policy, store, secrets));
        let session = Arc::new(KernelSession::new(handler.clone())?);
        Ok(Self { handler, session })
    }
}
```

All three dependencies are injected as trait objects. To swap in a different
store (say, the PostgreSQL backend instead of the in-memory one), the caller
passes a different `Arc<dyn ResourceStore>` and nothing else changes. There is
no conditional compilation, no feature flag, and no generic parameter on
`ServerHandler` itself.

## The Provides* traits

Three auxiliary traits expose the dependencies to downstream code:

```rust
pub trait ProvidesPolicy: Send + Sync + 'static {
    fn policy(&self) -> &Arc<dyn Policy>;
}

pub trait ProvidesResourceStore: Send + Sync + 'static {
    fn store(&self) -> &dyn ResourceStore;
}

pub trait ProvidesSecretManager: Send + Sync + 'static {
    fn secret_manager(&self) -> &dyn SecretManager;
}
```

`ServerHandlerInner` and `ServerHandler` both implement all three:

```rust
impl ProvidesPolicy for ServerHandler {
    fn policy(&self) -> &Arc<dyn Policy> { &self.handler.policy }
}

impl ProvidesResourceStore for ServerHandler {
    fn store(&self) -> &dyn ResourceStore { self.handler.store.as_ref() }
}

impl ProvidesSecretManager for ServerHandler {
    fn secret_manager(&self) -> &dyn SecretManager { self.handler.secrets.as_ref() }
}
```

The key design move is that the `store.rs` module defines blanket
`ResourceStore` and `ResourceStoreReader` implementations for any type that
implements `ProvidesResourceStore`:

```rust
#[async_trait]
impl<T: ProvidesResourceStore> ResourceStore for T {
    async fn create(&self, resource: Resource) -> Result<(Resource, ResourceRef)> {
        self.store().create(resource).await
    }
    // … all other methods delegate to self.store()
}
```

The same pattern applies to `Policy` and `SecretManager`. This means
`ServerHandler` itself satisfies `ResourceStore`, `Policy`, and `SecretManager`
— not by containing logic, but by delegation. Handler implementations can
call `self.create(resource)` directly on the handler type without needing to
write `self.store().create(resource)` everywhere.

## How handlers consume the composition

The generated handler traits have bounds on the `Provides*` traits indirectly
through the store/policy traits. An `impl CatalogHandler for ServerHandler` can
call `self.get(&ident).await` because `ServerHandler: ResourceStore` (via the
blanket impl). It can call `self.policy().check_required(...)` because
`ServerHandler: ProvidesPolicy`. Everything composes through the same mechanism.

For example, the catalog list handler calls `process_resources` from the policy
module, which requires `T: Policy`. `ServerHandler` satisfies that bound because
it implements `ProvidesPolicy`, which triggers the blanket `Policy` impl.

## TableManager and KernelSession

Table reads require more than the `ResourceStore`: they need the delta-kernel
session to load a Delta table snapshot from object storage. This is expressed
through a separate trait:

```rust
#[async_trait]
impl TableManager for ServerHandler {
    async fn read_snapshot(
        &self,
        location: &StorageLocationUrl,
        format: &DataSourceFormat,
        version: Option<Version>,
    ) -> Result<Arc<Snapshot>> {
        self.session.read_snapshot(location, format, version).await
    }
}
```

`KernelSession` handles the delta-kernel integration: it resolves object store
credentials (via `SecretManager`), constructs the appropriate `ObjectStore`
adapter for the target storage URL, and invokes delta-kernel to load the
snapshot. Handlers that read table data use `TableManager`; handlers that only
read metadata use `ResourceStore`.

## Adding a new backend

To add a new storage or policy backend:

1. Implement `ResourceStore` (or `Policy`, or `SecretManager`) on your type.
2. Wrap an instance in `Arc` and pass it to `ServerHandler::try_new_tokio`.

No handler code needs to change. The `Provides*` + blanket-impl pattern means
the new implementation is wired in at the single construction site. There is no
registry, no plugin system, and no runtime type dispatch beyond the vtable
already implied by `dyn ResourceStore`.

## Why not generics on ServerHandler?

An alternative design would make `ServerHandler` generic over its dependencies:

```rust
// hypothetical — not the actual design
pub struct ServerHandler<P: Policy, S: ResourceStore, M: SecretManager> {
    policy: Arc<P>,
    store:  Arc<S>,
    secrets: Arc<M>,
}
```

This would eliminate the vtable dispatch on every store call at the cost of
monomorphisation: every concrete combination of `(P, S, M)` would produce a
separate compiled type, and the handler traits would need to carry the generic
parameters too. More importantly, Axum requires its state type to be `Clone +
Send + Sync + 'static`; a generic `ServerHandler<P, S, M>` would propagate type
parameters throughout the router definition, making it awkward to compose
multiple services.

The `Arc<dyn Trait>` design was chosen because the indirection cost is
negligible for metadata operations (which are predominantly I/O bound), the
compilation unit stays small, and the router wiring stays simple.
