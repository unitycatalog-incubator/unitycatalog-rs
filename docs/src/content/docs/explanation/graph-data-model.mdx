---
title: The Graph Data Model
description: >
  Why unitycatalog-rs stores all resources as Object nodes and Association edges
  instead of a relational schema.
---

Every resource managed by the Unity Catalog server — Catalog, Schema, Table,
Volume, Share, Recipient, Credential, ExternalLocation, Column — is stored using
the same two primitives: **Object nodes** and **Association edges**. This page
explains why that design was chosen and what trade-offs it brings.

## The two primitives

### Object nodes

In code, a node in the store is either a typed `Resource` enum variant or the
lower-level `Object` struct that backs the PostgreSQL and in-memory stores:

```rust
pub enum Resource {
    Share(Share),
    Credential(Credential),
    Catalog(Catalog),
    Schema(Schema),
    Table(Table),
    ExternalLocation(ExternalLocation),
    Recipient(Recipient),
    Column(Column),
}

pub struct Object {
    pub id: Uuid,
    pub label: ObjectLabel,
    pub name: ResourceName,
    pub properties: Option<serde_json::Value>,
    pub created_at: chrono::DateTime<chrono::Utc>,
    pub updated_at: Option<chrono::DateTime<chrono::Utc>>,
}
```

`ObjectLabel` is the discriminant that identifies what kind of resource a node
represents. The full set of variants is:

```rust
// ObjectLabel variants (from crates/common/src/models/mod.rs)
ObjectLabel::Share
ObjectLabel::Credential
ObjectLabel::Catalog
ObjectLabel::Schema
ObjectLabel::Table
ObjectLabel::ExternalLocation
ObjectLabel::Recipient
ObjectLabel::Column
```

The typed `Resource` enum and the flat `Object` struct are interconvertible
through `TryFrom` impls generated by the `object_conversions!` macro. The
properties field of `Object` carries the resource-specific fields serialised as
JSON; the label tells the store which concrete type to deserialise into.

### Association edges

Relationships between nodes are stored as labelled, directed edges. Every edge
has a source, a target, an `AssociationLabel`, and optional property metadata:

```rust
pub enum AssociationLabel {
    OwnedBy,
    OwnerOf,
    DependsOn,
    DependencyOf,
    ParentOf,
    ChildOf,
    HasPart,
    PartOf,
    References,
    ReferencedBy,
}
```

Associations are always bidirectional: when the store records `A --ChildOf--> B`
it automatically records the inverse `B --ParentOf--> A`. The `inverse()` method
on `AssociationLabel` encodes that symmetry:

```rust
AssociationLabel::ChildOf => Some(AssociationLabel::ParentOf),
AssociationLabel::ParentOf => Some(AssociationLabel::ChildOf),
AssociationLabel::OwnedBy => Some(AssociationLabel::OwnerOf),
// …
```

Between any two resources there can be at most one association with a given
label; attempting to add a duplicate returns `Error::AlreadyExists`.

## Hierarchy as associations

In a conventional relational design, the hierarchy Catalog → Schema → Table
would be expressed as foreign keys: a schema row would have a `catalog_id`
column, a table row would have a `schema_id` column, and so on. Adding a new
nesting level would require a schema migration.

In the graph model the same hierarchy is expressed as associations. A Schema is
a node with `ObjectLabel::Schema`; its containment inside a Catalog is a
`ChildOf` edge from the Schema to the Catalog (equivalently, a `ParentOf` edge
from the Catalog to the Schema). Listing all schemas inside a catalog is a graph
traversal: `list_associations(catalog_id, AssociationLabel::ParentOf, ...)`.

This means adding a new containment relationship — say, linking a Column to a
Table — requires no DDL change to the backing store: you create a Column node
and add a `ChildOf` edge. The store schema is stable regardless of how many
resource types or nesting levels are introduced.

## The ResourceStore trait

All store operations are expressed through two traits:

```rust
#[async_trait]
pub trait ResourceStoreReader: Send + Sync + 'static {
    async fn get(&self, id: &ResourceIdent) -> Result<(Resource, ResourceRef)>;
    async fn get_many(&self, ids: &[ResourceIdent]) -> Result<Vec<(Resource, ResourceRef)>>;
    async fn list(
        &self,
        label: &ObjectLabel,
        namespace: Option<&ResourceName>,
        max_results: Option<usize>,
        page_token: Option<String>,
    ) -> Result<(Vec<Resource>, Option<String>)>;
}

#[async_trait]
pub trait ResourceStore: ResourceStoreReader {
    async fn create(&self, resource: Resource) -> Result<(Resource, ResourceRef)>;
    async fn delete(&self, id: &ResourceIdent) -> Result<()>;
    async fn update(&self, id: &ResourceIdent, resource: Resource) -> Result<(Resource, ResourceRef)>;

    async fn add_association(
        &self,
        from: &ResourceIdent,
        to: &ResourceIdent,
        label: &AssociationLabel,
        properties: Option<PropertyMap>,
    ) -> Result<()>;

    async fn remove_association(
        &self,
        from: &ResourceIdent,
        to: &ResourceIdent,
        label: &AssociationLabel,
    ) -> Result<()>;

    async fn list_associations(
        &self,
        resource: &ResourceIdent,
        label: &AssociationLabel,
        target_label: Option<&ResourceIdent>,
        max_results: Option<usize>,
        page_token: Option<String>,
    ) -> Result<(Vec<ResourceIdent>, Option<String>)>;
}
```

`ResourceIdent` carries both the label and a `ResourceRef` (either a UUID or a
namespaced name). The store is responsible for assigning globally unique IDs on
creation; callers must not assume the ID they provide will be used.

Both the in-memory store (`crates/server/src/memory.rs`) and the PostgreSQL
backend (`crates/postgres/`) implement this trait. Higher-level code only ever
touches `dyn ResourceStore`.

## Relationship to Meta's TAO paper

The design deliberately mirrors the **TAO (The Associations and Objects)**
storage model described by Facebook/Meta. TAO models the social graph as a set
of typed objects (nodes) and typed associations (edges). The key insight is that
the graph model decouples the *shape* of relationships from the backing storage
engine: you can add new association types without touching the store schema, and
you can answer questions that span multiple resource types through graph
traversal rather than SQL joins.

unitycatalog-rs applies the same pattern to the metadata catalog graph: catalogs,
schemas, tables, shares, and recipients are all objects; containment, ownership,
and dependency relationships are associations.

## Trade-offs

| Concern | Relational schema | Graph model (this project) |
|---------|-------------------|---------------------------|
| Adding a new resource type | New table + migration | New `ObjectLabel` variant, no DDL |
| Adding a new relationship | New FK column or join table + migration | New `AssociationLabel` variant, no DDL |
| Point lookup by id | O(1) primary-key lookup | O(1) primary-key lookup (same) |
| Hierarchy traversal | Multi-table JOIN | Multi-hop `list_associations` call |
| Schema rigidity | High — FK constraints enforce shape | Low — label validation is in Rust code |
| Query expressiveness | Full SQL | Limited to the `ResourceStore` API surface |

The last point is the main constraint: complex reporting queries (e.g. "find all
tables with a dependency on credential X that are also owned by user Y") are not
first-class in the current API. If such queries become important, they would be
expressed as a series of `list_associations` calls and intersected in application
code, or handled via a dedicated query layer on top of the PostgreSQL backend.
