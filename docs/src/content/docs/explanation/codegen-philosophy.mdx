title: Code Generation Philosophy
description: >
  How Protocol Buffer definitions drive a single-source-of-truth pipeline that
  generates Axum handlers, HTTP clients, and language bindings.

The API surface of unitycatalog-rs is defined once, in Protocol Buffers, and
then mechanically expanded into Rust server code, a Rust HTTP client, Python
bindings, and TypeScript/Node.js bindings. This page explains why that
approach was chosen and how the pipeline is structured.

## Why proto files are the source of truth

The Unity Catalog REST API is large: catalogs, schemas, tables, volumes, shares,
recipients, credentials, external locations, temporary credentials — each with
CRUD operations and pagination. Writing all of that by hand, consistently, in
four different languages, and keeping them in sync across API changes, would be
error-prone and expensive.

Protocol Buffers solve the consistency problem. A `.proto` file precisely
describes every request message, response message, field type, optionality, and
HTTP binding. That description is machine-readable, so a code generator can
derive all the boilerplate from it deterministically.

Doc comments in proto files are propagated into the generated Rust code (added
in commit `e6b9eab`). This means API documentation only needs to be written in
one place and it appears in the generated handler traits, client builders, and
language bindings automatically.

## Pipeline structure

```
proto/
  *.proto              ← authoritative API definition
     │
     ▼
buf compile            ← produces a binary FileDescriptorSet
     │
     ▼
crates/build           ← internal code generator binary
  ├── parsing/         ← deserialise FileDescriptorSet into CodeGenMetadata
  ├── analysis/        ← build a ServicePlan / MethodPlan for each service
  └── codegen/         ← emit Rust (and TypeScript) from the plan
         │
         ├── output_common  → crates/common/src/codegen/
         ├── output_server  → crates/server/src/codegen/
         ├── output_client  → crates/client/src/codegen/
         ├── output_python  → python/client/src/codegen/
         ├── output_node    → node/client/src/codegen/   (NAPI)
         └── output_node_ts → node/client/src/           (TypeScript)
```

The generator is invoked via `just generate` (full pipeline including external
types) or `just generate-code` (internal pipeline only). It reads a compiled
`FileDescriptorSet` binary rather than parsing `.proto` text directly, so it
benefits from `buf`'s import resolution and validation.

## What gets generated

### Axum handler traits (server)

For each proto service the generator emits an async trait. The `CatalogHandler`
trait, for example, looks like:

```rust
// @generated — do not edit by hand.
#[async_trait]
pub trait CatalogHandler: Send + Sync + 'static {
    /// List catalogs
    ///
    /// Gets an array of catalogs in the metastore. …
    async fn list_catalogs(
        &self,
        request: ListCatalogsRequest,
        context: RequestContext,
    ) -> Result<ListCatalogsResponse>;

    async fn create_catalog(
        &self,
        request: CreateCatalogRequest,
        context: RequestContext,
    ) -> Result<Catalog>;

    // … get, update, delete
}
```

The doc comments come directly from the proto file. A hand-written
`impl CatalogHandler for ServerHandler` in `crates/server/src/api/catalogs.rs`
provides the business logic; the generated file only defines the contract.

The generator also emits Axum route handler functions and `FromRequest`
extractors that call the trait methods, so the implementer never writes HTTP
plumbing.

### HTTP client (client crate)

The generator emits a typed builder-pattern client for each service. Callers
construct requests with strongly typed fields and the client handles
serialisation, URL building, and pagination.

### Python bindings (PyO3)

The generator emits `#[pyclass]` and `#[pymethods]` wrappers that expose the
same request/response types to Python. Maturin compiles them into a wheel.

### Node.js bindings (NAPI + TypeScript)

Two outputs are produced for Node.js: a NAPI Rust shared library that exports
native bindings, and a TypeScript client that wraps the NAPI layer with types
and ergonomic promise-based APIs.

## Anatomy of a generated file

All generated files begin with:

```rust
// @generated — do not edit by hand.
```

This marker means editors and reviewers can immediately identify that the file
should not be manually modified. Changes to API shape, field names, or doc
comments must be made in the corresponding `.proto` file and regenerated.

Generated files are committed to the repository (they are not `.gitignore`d).
This keeps the repository self-contained: a contributor can build and test
without running the full generation pipeline, and CI can detect when generated
files are out of sync with the proto definitions.

## Adding a new endpoint

The workflow for extending the API is:

1. Add the new RPC and its request/response messages to the appropriate `.proto`
   file in `proto/`.
2. Run `just generate` to regenerate all output files.
3. Implement the new method in the corresponding hand-written `impl` block
   (e.g. `crates/server/src/api/catalogs.rs` for catalog operations). The
   generated trait will produce a compile error until the method is implemented.
4. Add any new `ResourceStore` operations needed to fulfil the request.

Steps 1 and 2 are mechanical; the compiler enforces step 3 because the trait
method is non-optional.

## Trade-offs of generated code

The approach trades flexibility for consistency. Custom middleware, non-standard
HTTP semantics, or endpoints that do not fit the proto-to-REST mapping
require either extending the generator or writing the handler outside the
generated layer. The sharing endpoints (`crates/server/src/sharing/`) are an
example of the latter: they implement a separate protocol (Delta Sharing) and
have their own router rather than going through the standard codegen path.
