The in-memory server from the [first tutorial](/tutorials/01-first-server/) loses all data when it stops. This tutorial switches to a PostgreSQL backend so that catalogs, schemas, and tables survive server restarts.

## Prerequisites

- **Docker** and **Docker Compose** — install via [Docker Desktop](https://www.docker.com/products/docker-desktop/) or your OS package manager
- **[just](https://just.systems/)** — the task runner
- **curl** and **jq** — for verifying API responses

Verify Docker is running:

```bash
docker info
```

## How it works

The `dev/` directory in the repository contains Docker Compose configuration that starts a PostgreSQL instance alongside the Unity Catalog server. The `just rest-db` command brings up that Compose stack and starts the server configured to use the local PostgreSQL instance.

The server configuration for the PostgreSQL backend looks like this:

```json
{
  "host": "0.0.0.0",
  "port": 8080,
  "backend": {
    "engine": "postgres",
    "database": "postgres",
    "host": "localhost",
    "port": "5432",
    "user": "postgres",
    "password": {"env": "POSTGRES_PASSWORD"}
  }
}
```

The `password` field uses `{"env": "POSTGRES_PASSWORD"}` to read the password from an environment variable rather than hardcoding it in the config file.

## Step 1 — Start the PostgreSQL-backed server

```bash
just rest-db
```

This command:
1. Starts the PostgreSQL container via Docker Compose (from `dev/`)
2. Waits for the database to be ready
3. Runs any pending database migrations
4. Starts the Unity Catalog server on `http://localhost:8080`

Wait until you see output like:

```
Listening on 0.0.0.0:8080
```

<Aside type="note">
The first run also pulls the PostgreSQL Docker image, so it may take a minute longer than subsequent starts.
</Aside>

## Step 2 — Create a catalog

Open a second terminal and create a catalog using the same REST API as before:

```bash
curl -s -X POST http://localhost:8080/api/2.1/unity-catalog/catalogs \
  -H "Content-Type: application/json" \
  -d '{"name": "persistent_catalog"}' | jq .
```

Expected response:

```json
{
  "name": "persistent_catalog",
  "comment": null,
  "properties": {},
  "created_at": 1700000000000,
  "updated_at": null,
  "id": "..."
}
```

Confirm it exists:

```bash
curl -s http://localhost:8080/api/2.1/unity-catalog/catalogs | jq '.catalogs[].name'
```

```
"persistent_catalog"
```

## Step 3 — Stop the server

Switch back to the terminal running `just rest-db` and press **Ctrl-C**. This stops the Unity Catalog server process. The PostgreSQL container continues running in the background.

## Step 4 — Restart the server

```bash
just rest-db
```

Wait for the `Listening on 0.0.0.0:8080` message again.

## Step 5 — Verify data persisted

```bash
curl -s http://localhost:8080/api/2.1/unity-catalog/catalogs | jq '.catalogs[].name'
```

You should see:

```
"persistent_catalog"
```

The catalog survived the restart because it is stored in PostgreSQL, not in memory.

## Step 6 — Create a schema and table

Continue building out the hierarchy:

```bash
curl -s -X POST http://localhost:8080/api/2.1/unity-catalog/schemas \
  -H "Content-Type: application/json" \
  -d '{"catalog_name": "persistent_catalog", "name": "main"}' | jq .
```

```bash
curl -s -X POST http://localhost:8080/api/2.1/unity-catalog/tables \
  -H "Content-Type: application/json" \
  -d '{
    "catalog_name": "persistent_catalog",
    "schema_name": "main",
    "name": "events",
    "table_type": "EXTERNAL",
    "data_source_format": "DELTA",
    "columns": []
  }' | jq .
```

Stop and restart the server again (`Ctrl-C`, then `just rest-db`) and verify the schema and table are still present:

```bash
curl -s "http://localhost:8080/api/2.1/unity-catalog/schemas?catalog_name=persistent_catalog" | jq '.schemas[].full_name'
curl -s "http://localhost:8080/api/2.1/unity-catalog/tables?catalog_name=persistent_catalog&schema_name=main" | jq '.tables[].full_name'
```

## Shut down everything

To stop both the server and the PostgreSQL container:

```bash
# Ctrl-C to stop the server, then:
cd dev && docker compose down
```

<Aside type="caution">
Running `docker compose down` removes the containers but by default preserves the named Docker volume containing PostgreSQL data. To also delete all stored data, run `docker compose down -v`.
</Aside>
